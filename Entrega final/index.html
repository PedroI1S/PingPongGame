Mergulho 1 · Física

Por que reconstruir a física

Antes (parcial)

  • Quique = vy ← −0.7·vy
  • Sem spin — bola nunca curva
  • Bot acertava por sorteio de dado
  • Saque com velocidade fixa

Depois

  • Integrador único: gravidade, arrasto, Magnus
  • Quique acopla spin ↔ velocidade
  • Bot prevê a trajetória; erra como humano
  • Clique decide mira, ritmo e efeito

A jogada deixou de ser "acertar o clique" e virou controlar o efeito.

(0:50) Contraste honesto: o que tinha antes era um placeholder. Liste as 4 limitações, e diga que a meta foi spin de verdade com consequência (curva + quique diferente).

Mergulho 1 · Física

O integrador demo

Ajuste o spin e ligue/desligue o Magnus
WebGL indisponível — a curva de Magnus surge ao aplicar spin: a bola desvia na direção de ω × v.

Euler semi-implícito, substeps fixos 240 Hz + acumulador → determinístico e independente de FPS.

// substep fixo (dt = 1/240 s)
a.set(0, -g, 0);
a.mulAdd(vel, -dragK*vel.len()); // arrasto
m.set(spin).crs(vel).scl(kMag);  // ω × v
a.add(m);
vel.mulAdd(a, dt);   // Euler semi-impl.
pos.mulAdd(vel, dt);

Correlação — aula "Câmeras, delta e renderização": o professor ensina multiplicar por delta para normalizar movimento entre FPS. Para física isso não basta (não é determinístico) — então fui além com fixed-timestep + acumulador.

(1:40) Rode a demo: comece sem spin (reta-ish), depois aumente o spin → a curva aparece. Topspin mergulha, sidespin desvia lateral. Explique a equação a = k(ω×v). O ponto da CORRELAÇÃO é forte: delta da aula resolve animação; física precisa de passo fixo. Q: "por que 240 Hz?" → divide 60 fps certinho e dá estabilidade ao integrador.
Mergulho 1 · Física

Quique com efeito & rede de verdade

Quique na mesa

  • Vertical: vy ← −e·vy, e=0.92
  • Horizontal: impulso de atrito opõe o escorregamento no contato → acopla spin↔velocidade
  • Topspin avança, backspin morre, sidespin desvia
  • Inércia de esfera 2/5·r², cap de aderência 2/7

Rede = colisor varrido (swept)

  • Testo o segmento anterior→atual no cruzamento z=0 — sem tunneling mesmo na velocidade máxima
  • Contato mata 80–90% da velocidade + jitter de corda
  • Bola pinga por cima ou volta; as regras pontuam o resultado

Por que swept: a 60 fps a bola anda vários "raios" por frame; um teste pontual deixaria ela atravessar a rede. Detecção contínua resolve.

(1:10) Coração do "efeito tem consequência": o mesmo impulso muda velocidade E spin no quique. Topspin pega pra frente, backspin trava. Swept collider = anti-tunneling — bom ponto técnico. Q: "let (toque na rede)?" → optei por jogar sem regra de let; é mais simples e divertido.
Mergulho 1 · Física

Onde você clica importa

O offset do clique sobre a bola (ndx, ndy ∈ [−1,1]) vira mira, efeito e ritmo:

  • Lateral → mira + sidespin (curva tipo hook)
  • Cima = topspin (mergulha) · baixo = backspin (flutua)
  • Ritmo carrega parte da velocidade que chega
  • Clamps no fim = envelope legal (e anti-cheat)
// contato fora do centro → mira, spin, ritmo
spinVert =  ndx * sidespinGain;   // curva lateral
spinLat  = -ndy * topspinGain;    // cima=top, baixo=back
vForward = basePace*(1 + .25f*off)
         + .35f*vIn.len();        // + carry do que chega
clampSpeedAndSpin(out);           // teto físico = validação

O mesmo mapeamento vale para saque e para a rebatida do bot.

(1:00) Esta é a "mira analógica de graça": borda direita = vai pra direita + curva. Destaque que serve, retorno e bot passam todos pela MESMA função → consistência. Os clamps no fim são reaproveitados como validação de rede (gancho pro mergulho de anti-cheat).
Mergulho 2 · Redes

Servidor autoritativo demo

  • O GameServer roda headless: simula a 60 Hz e faz broadcast do estado a 30 Hz
  • Vive como daemon dentro do host ou como servidor dedicado
  • Clientes só desenham e mandam input — nunca decidem a partida
while (running) {                 // loop de passo fixo
  world.update(delta);            // simula a 60 Hz
  stateAcc += delta;
  if (stateAcc >= 1f/30f) {       // envia a 30 Hz
    broadcastState(); stateAcc = 0;
  }
  parkNanos(TICK_NS - elapsed);
}
Servidor a 60 Hz; envia STATE a 30 Hz; o cliente extrapola a curva entre os pontos.

Aula "Comunicação pela rede": sockets Java + thread por conexão. Degrau acima: de um exemplo request/response para um loop autoritativo em tempo real.

(1:40) A ideia central: UMA fonte de verdade. Servidor simula a 60, envia a 30 (metade — economia de banda). Mostre na timeline: ticks densos do servidor vs pontos de STATE espaçados. Q: "por que 30 e não 60?" → banda; e o cliente preenche o meio (próximo do dead reckoning). Q: "TCP ou UDP?" → TCP: em LAN a ordem/garantia compensa e simplifica.

RoomCode = IPv4 do host em base-36 (código de 7 caracteres em vez de "digite o IP").

Autoral vs a aula: o professor usa ObjectOutputStream (serialization Java). Troquei por um protocolo binário escrito à mão (DataInputStream + byte de tipo): menor, mais rápido, sem reflexão.

Anti-cheat: o cliente só clica demo

Clique na bola (ou use o botão)
O cliente manda só (x,y); o servidor reconstrói o raio da câmera e testa o acerto.
// cliente: envia SÓ a tela
send(CLICK, x, y, vw, vh);

// servidor refaz o MESMO raio
Ray r = ServerPickRay
        .fromScreen(p, x, y, vw, vh);
// e valida o acerto
boolean hit = Intersector
   .intersectRaySphere(r, ball, rad);

ServerPickRay é Java puro (sem JNI) → roda no servidor headless. Pacotes SERVE/HIT antigos foram aposentados.

Aula de Câmeras (project/unproject para hitboxes): aqui o unproject é refeito no servidor — o cliente não consegue forjar uma velocidade impossível.

Mergulho 2 · Redes

Suavização entre snapshots

O cliente renderiza a 60+ fps, mas só recebe STATE a 30 Hz. Entre eles ele extrapola com a MESMA física:

// dead reckoning entre snapshots de 30 Hz
ball.set(lastSnapshot);            // pos + vel + spin
physics.step(ball, delta, null, contacts);
// null = sem jitter → resultado determinístico

Como o STATE carrega o spin, a curva no cliente bate exatamente com a do servidor.

Decisão consciente

É extrapolação (dead reckoning) — não client-side prediction com rollback.

Em LAN a latência é < 5 ms: o ganho de prediction é quase nulo e o custo de reconciliação é alto. Para WAN, eu revisitaria.

Reúso elegante: servidor, bot e cliente usam o mesmo BallPhysics. Uma física, três usos.

(1:10) Conecta os mergulhos: a MESMA classe de física do mergulho 1 roda no cliente para preencher o meio dos snapshots. Seja honesto: é dead reckoning, não prediction. Justifique por LAN. Q: "e se um STATE chega fora de ordem?" → TCP garante ordem; em UDP eu usaria sequence number.

Como tudo se sustenta

Módulos & testes headless

Quatro módulos com fronteiras claras:

  • sim — física + protocolo + servidor, sem UI libGDX
  • core — cliente (telas, render, shader)
  • server — só o main() do dedicado
  • lwjgl3 — launcher desktop

Como o sim não tem janela, ele roda no JVM headless e é testado com JUnit.

A suíte fixa o comportamento

  • Voo, quique, rede, contato da raquete
  • Determinismo: mesma seed → trajetória bit-idêntica
  • Bot preditivo calibrado por Monte-Carlo (retorno ≈ 0,73) — substituiu o sorteio

Aula "Inicialização, módulos e ciclo de vida": a ideia de módulos e do starter levada a um módulo de simulação isolado e testável — é o que deixa o servidor headless e o anti-cheat possíveis.

(1:00) Aqui entra o bot que cortamos do slide próprio: ele agora é PREDITIVO e está FIXADO por teste Monte-Carlo + determinismo. O grande ponto: separar 'sim' sem UI foi o que permitiu testar física a sério E rodar o servidor headless. Correlação com a aula de módulos/ciclo de vida.
