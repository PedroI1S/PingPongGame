Resposta da atividade



Nesta atividade foi desenvolvida, em libGDX, uma tela de configuração que reúne elementos de formulário e um mecanismo de save. A troca de telas é feita pela interface Screen, e os campos do formulário (toggle, slider e radio) são ligados diretamente aos dados persistidos, de forma orientada a objetos e com baixo acoplamento. A tela de loading não foi usada como entrega, conforme pedido.



1. Tela de formulário com a interface Screen



A tela de configuração é a ConfigScreen, que estende BaseScreen. A BaseScreen, por sua vez, estende ScreenAdapter da libGDX, ou seja, implementa a interface Screen (com os métodos show, render, resize, hide e dispose). No show() o input é registrado e os campos do formulário são montados; no hide() o input é liberado.



Exemplo do ciclo de vida da tela (Screen):



public final class ConfigScreen extends BaseScreen { 
 
    @Override 
    public void show() { 
        Gdx.input.setInputProcessor(input); 
        rebuildTabContent();   // monta os campos do formulario 
    } 
 
    @Override 
    public void hide() { 
        Gdx.input.setInputProcessor(null); 
    } 
}


2. Elementos de formulário



Os campos são widgets próprios (sem usar o Scene2D), cada um stateless: recebe um reader para ler o valor atual e um writer para gravar o novo valor. São três tipos clássicos de formulário: Toggle (liga/desliga), Slider (faixa numérica) e Radio (seleção exclusiva).



Exemplo do Toggle (campo booleano):



public boolean tryClick(float wx, float wy) { 
    if (!bounds.contains(wx, wy)) return false; 
    writer.accept(!reader.getAsBoolean());   // inverte e grava 
    return true; 
}


Exemplo do Slider, convertendo a posição do cursor em valor:



private void applyAt(float wx) { 
    float t = (wx - bounds.x) / bounds.width; 
    t = Math.max(0f, Math.min(1f, t)); 
    float raw = min + t * (max - min); 
    int snapped = (int) (Math.round(raw / step) * step); 
    writer.accept(snapped);   // grava no GameSettings 
}


Exemplo do Radio (seleção exclusiva):



public boolean tryClick(float wx, float wy) { 
    if (!bounds.contains(wx, wy)) return false; 
    int idx = (int) ((wx - bounds.x) / pillWidth); 
    if (idx < 0 || idx >= options.length) return false; 
    writer.accept(options[idx]);   // seleciona a opcao 
    return true; 
}


3. Troca de tela com a interface Screen



A navegação é centralizada na classe Main (um Game da libGDX), que troca a tela atual com setScreen(...). O menu chama openConfig() para abrir o formulário.



Exemplo da troca de tela:



public void openConfig() { 
    setScreen(new ConfigScreen(this)); 
}


4. Ligação dos campos aos dados



Ao montar a aba, cada campo é "amarrado" aos getters e setters de GameSettings usando referências de método. É aqui que o formulário se conecta ao save.



Exemplo da ligação dos campos:



// slider de volume ligado ao get/set do SFX 
sliders.add(makeVolSlider(sliderX, y, w, s::getSfxVolume, s::setSfxVolume)); 
 
// toggles ligados direto aos get/set (que ja salvam) 
toggles.add(new Toggle(toggleX, y, s::isShowFpsCounter, s::setShowFpsCounter)); 
toggles.add(new Toggle(toggleX, y, s::isScreenShakeEnabled, s::setScreenShakeEnabled));


5. Mecanismo de save com Preferences



A persistência usa Preferences da libGDX, o armazenamento chave/valor que grava em disco. O GameSettings lê tudo no boot com load() e regrava com save() seguido de flush(). O load() é chamado no construtor do GameContext, então as escolhas do jogador voltam ao reabrir o jogo.



Exemplo de leitura e gravação:



public void load() { 
    Preferences prefs = Gdx.app.getPreferences(PREFS_NAME); 
    sfxVolume = prefs.getInteger(KEY_SFX_VOL, sfxVolume); 
    showFpsCounter = prefs.getBoolean(KEY_FPS_COUNTER, showFpsCounter); 
} 
 
public void save() { 
    Preferences prefs = Gdx.app.getPreferences(PREFS_NAME); 
    prefs.putInteger(KEY_SFX_VOL, sfxVolume); 
    prefs.putBoolean(KEY_FPS_COUNTER, showFpsCounter); 
    prefs.flush();   // persiste em disco 
}


Carregamento do save no boot, dentro do GameContext:



public GameContext() { 
    // ... fontes, viewport, texturas ... 
    settings.load();   // carrega o save logo no inicio 
}


6. Save automático ao alterar o formulário



Não existe botão "Salvar": cada setter chama save() na hora, então mexer em qualquer campo já grava o valor em disco (a barra inferior mostra "CHANGES SAVED AUTOMATICALLY").



Exemplo dos setters com save automático:



public void setSfxVolume(int v) { sfxVolume = clampVol(v); save(); } 
public void setShowFpsCounter(boolean v) { showFpsCounter = v; save(); } 
public void setScreenShakeEnabled(boolean v) { screenShake = v; save(); }
